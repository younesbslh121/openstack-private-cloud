\chapter*{R\'esum\'e}
\addcontentsline{toc}{chapter}{R\'esum\'e}

Ce projet traite de la conception, du d\'eploiement et de la supervision d'un environnement \textbf{Cloud Priv\'e} bas\'e sur la plateforme Open Source \textbf{OpenStack}. L'objectif principal est de mettre en \oe{}uvre une infrastructure compl\`ete d'Infrastructure as a Service (IaaS), automatis\'ee de bout en bout.

L'originalit\'e de ce travail r\'eside dans l'utilisation de m\'ethodes d'automatisation de pointe : \textbf{Kolla-Ansible} pour le d\'eploiement conteneuris\'e int\'egral des micro-services OpenStack (38 conteneurs Docker), et \textbf{Terraform} pour le provisionnement d\'eclaratif de l'infrastructure virtuelle (r\'eseaux, instances, groupes de s\'ecurit\'e).

Un syst\`eme de supervision continue a \'et\'e impl\'ement\'e via \textbf{Prometheus} et \textbf{Grafana}, permettant le monitoring en temps r\'eel des m\'etriques syst\`eme, des conteneurs Docker, et des services OpenStack.

\vspace{0.5cm}
\noindent\textbf{Mots-cl\'es :} Cloud Priv\'e, OpenStack, Kolla-Ansible, Terraform, Infrastructure as Code, Docker, Conteneurisation, Prometheus, Grafana, Haute Disponibilit\'e, DevOps, IaaS.

% --- Abstract ---
\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

This project addresses the design, deployment, and monitoring of a \textbf{Private Cloud} environment based on the open-source \textbf{OpenStack} platform. The primary goal is to implement a complete Infrastructure as a Service (IaaS) environment, fully automated from end to end.

The originality lies in leveraging cutting-edge automation tools: \textbf{Kolla-Ansible} for fully containerized microservice deployment (38 Docker containers), and \textbf{Terraform} for declarative infrastructure provisioning (virtual networks, compute instances, and security groups).

A continuous monitoring system was implemented using \textbf{Prometheus} and \textbf{Grafana}, enabling real-time monitoring of system metrics, Docker containers, and OpenStack services.

\vspace{0.5cm}
\noindent\textbf{Keywords:} Private Cloud, OpenStack, Kolla-Ansible, Terraform, Infrastructure as Code, Docker, Containerization, Prometheus, Grafana, High Availability, DevOps, IaaS.

\chapter{Introduction G\'en\'erale}

\section{Contexte et Motivation}

\`A l'\`ere de la transformation num\'erique, la capacit\'e de fournir des ressources informatiques de mani\`ere agile, \'elastique et automatis\'ee est devenue un enjeu strat\'egique majeur pour les entreprises et les institutions acad\'emiques. L'\'evolution du concept de Cloud Computing a m\'etamorphos\'e la gestion des datacenters traditionnels en des plateformes virtuelles complexes et scalables.

Bien que les fournisseurs de Cloud public (AWS, Microsoft Azure, Google Cloud Platform) dominent le march\'e mondial, des contraintes strictes de \textbf{souverainet\'e des donn\'ees}, de \textbf{conformit\'e r\'eglementaire} (RGPD, loi marocaine 09-08), et de \textbf{ma\^itrise des co\^uts} \`a long terme poussent de nombreuses organisations \`a rapatrier leurs infrastructures au sein de Clouds Priv\'es. C'est dans ce contexte que s'inscrit l'utilisation d'OpenStack, la plateforme de Cloud Open Source la plus mature et la plus d\'eploy\'ee au monde.

Ce projet s'inscrit dans le cadre de la formation d'ing\'enieur \`a l'INPT, \`a la crois\'ee de l'administration syst\`eme, du r\'eseau, et du paradigme DevOps. Il vise \`a d\'emontrer la capacit\'e de concevoir, d\'eployer et superviser un environnement IaaS complet sur une infrastructure minimale.

\section{Probl\'ematique}

Comment d\'eployer un Cloud Priv\'e OpenStack complet de mani\`ere automatis\'ee, reproductible et supervis\'ee sur un environnement \`a ressources contraintes, tout en garantissant la haute disponibilit\'e des services et l'automatisation du provisionnement de l'infrastructure via l'Infrastructure as Code ?

\section{Objectifs du Projet}

\begin{enumerate}[leftmargin=2cm]
    \item \textbf{D\'eployer} un Cloud Priv\'e OpenStack complet via le framework Kolla-Ansible en mode All-in-One, avec conteneurisation int\'egrale des micro-services.
    \item \textbf{Automatiser} la cr\'eation des r\'eseaux virtuels, des groupes de s\'ecurit\'e et des instances de calcul via Terraform (Infrastructure as Code).
    \item \textbf{Instrumenter} la supervision continue de l'infrastructure via un stack de monitoring Prometheus/Grafana d\'ecoupl\'e.
    \item \textbf{Documenter} l'architecture de r\'ef\'erence et les proc\'edures op\'erationnelles pour faciliter la reproductibilit\'e.
\end{enumerate}

\section{M\'ethodologie Adopt\'ee}

La m\'ethodologie adopt\'ee suit le cycle de vie DevOps iteratif :
\begin{enumerate}
    \item \textbf{Phase de conception} : \'Etude de l'architecture cible, choix technologiques, dimensionnement.
    \item \textbf{Phase de d\'eploiement} : Automatisation via scripts Bash et Kolla-Ansible.
    \item \textbf{Phase de provisionnement} : Infrastructure as Code avec Terraform.
    \item \textbf{Phase de supervision} : Monitoring avec Prometheus, Grafana, cAdvisor, Node Exporter.
    \item \textbf{Phase de validation} : Tests fonctionnels et de performance.
\end{enumerate}

\chapter{\'Etat de l'Art et Fondements Technologiques}

\section{Le Paradigme du Cloud Computing}

\subsection{D\'efinition}
Le National Institute of Standards and Technology (NIST) d\'efinit le Cloud Computing comme un mod\`ele permettant un \textit{acc\`es r\'eseau omnipr\'esent, pratique et \`a la demande} \`a un ensemble partag\'e de ressources informatiques configurables (r\'eseaux, serveurs, stockage, applications et services) qui peuvent \^etre rapidement provisionn\'ees et lib\'er\'ees avec un effort de gestion minimal ou une interaction minimale avec le fournisseur de services.

\subsection{Caract\'eristiques Essentielles}
Le NIST identifie cinq caract\'eristiques fondamentales du Cloud Computing :
\begin{itemize}
    \item \textbf{Libre-service \`a la demande (On-demand self-service)} : L'utilisateur provisionne les ressources automatiquement, sans intervention humaine du c\^ot\'e du fournisseur.
    \item \textbf{Acc\`es r\'eseau large (Broad network access)} : Les services sont accessibles via des protocoles standards (HTTP, TCP/IP) depuis une vari\'et\'e de terminaux.
    \item \textbf{Mutualisation des ressources (Resource pooling)} : Les ressources physiques du fournisseur sont partag\'ees entre plusieurs clients (mod\`ele multi-tenant) avec une allocation dynamique.
    \item \textbf{\'Elasticit\'e rapide (Rapid elasticity)} : Les capacit\'es de calcul, de stockage et de r\'eseau peuvent \^etre augment\'ees ou r\'eduites dynamiquement en fonction de la charge.
    \item \textbf{Service mesur\'e (Measured service)} : La consommation des ressources est mesur\'ee, contr\^ol\'ee et factur\'ee de mani\`ere transparente.
\end{itemize}

\section{OpenStack : Plateforme de Cloud Open Source}

\subsection{Historique et Gouvernance}
OpenStack est n\'e en 2010 d'une collaboration entre la NASA (projet Nebula) et Rackspace. Aujourd'hui g\'er\'e par l'Open Infrastructure Foundation (OIF), il est utilis\'e par plus de 300 organisations mondiales, dont CERN, Walmart, Bloomberg et China Mobile, g\'erant plus de 25 millions de c\oe urs de calcul.

\subsection{Architecture en Micro-Services}
OpenStack n'est pas un logiciel monolithique mais un \textbf{agr\'egat de micro-services interd\'ependants} communiquant via des APIs REST. Chaque service est un projet ind\'ependant avec son propre cycle de release.

\section{Kolla-Ansible : Conteneurisation d'OpenStack}

\subsection{Probl\'ematique du D\'eploiement Traditionnel}
Historiquement, l'installation d'OpenStack (DevStack, PackStack) requ\'erait la configuration minutieuse de paquets Linux directement sur le syst\`eme h\^ote, provoquant fr\'equemment des conflits de biblioth\`eques bloquants (le probl\`eme connu sous le nom de \textit{Dependency Hell}). La mise \`a jour ou le rollback \'etaient particuli\`erement p\'erilleux.

\subsection{Solution Kolla-Ansible}
Le projet \textbf{Kolla-Ansible} r\'esout cette probl\'ematique en adoptant une approche fondamentalement diff\'erente :
\begin{enumerate}
    \item \textbf{Kolla} : Construit des images Docker optimis\'ees pour chaque micro-service OpenStack.
    \item \textbf{Ansible} : Orchestre le d\'eploiement, la configuration et la mise \`a jour de ces conteneurs de mani\`ere idempotente.
\end{enumerate}

Cette approche garantit :
\begin{itemize}
    \item \textbf{Isolation} : Chaque service fonctionne dans son propre conteneur avec ses d\'ependances.
    \item \textbf{Reproductibilit\'e} : Le m\^eme playbook produit le m\^eme r\'esultat \`a chaque ex\'ecution.
    \item \textbf{Rollback} : Un rollback consiste simplement \`a red\'emarrer l'ancien conteneur.
    \item \textbf{Scalabilit\'e horizontale} : Ajout de n\oe{}uds sans red\'eploiement complet.
\end{itemize}

\section{Terraform et l'Infrastructure as Code (IaC)}

\subsection{Principe}
L'Infrastructure as Code (IaC) est le processus de gestion et de provisionnement de l'infrastructure informatique \`a travers des fichiers de configuration plut\^ot que par des proc\'ed\'es manuels interactifs.

\subsection{Terraform}
\textbf{Terraform}, d\'evelopp\'e par HashiCorp, est un outil Open Source d'IaC qui utilise un langage d\'eclaratif (HCL --- HashiCorp Configuration Language). Son fonctionnement repose sur trois \'etapes :
\begin{enumerate}
    \item \texttt{terraform init} : Initialisation et t\'el\'echargement des providers.
    \item \texttt{terraform plan} : Calcul du diff\'erentiel entre l'\'etat d\'esir\'e et l'\'etat actuel.
    \item \texttt{terraform apply} : Application des changements n\'ecessaires uniquement.
\end{enumerate}

Le \textbf{State File} (\texttt{terraform.tfstate}) conserve l'\'etat actuel de l'infrastructure et permet \`a Terraform de n'appliquer que les diff\'erences (approche \textit{plan-apply}).

\subsection{Provider OpenStack}
Le provider \texttt{terraform-provider-openstack} permet \`a Terraform d'interagir avec les APIs Keystone, Nova, Neutron, Cinder et Glance pour provisionner des ressources sur un Cloud OpenStack.

\section{Supervision : Prometheus et Grafana}

\subsection{Prometheus}
\textbf{Prometheus} est un syst\`eme de monitoring et d'alerte Open Source. Il utilise un mod\`ele \textit{pull} : il interroge p\'eriodiquement les cibles (endpoints) pour collecter les m\'etriques au format \texttt{time-series}. Son langage de requ\^ete \textbf{PromQL} permet des analyses puissantes.

\subsection{Grafana}
\textbf{Grafana} est une plateforme de visualisation qui se connecte \`a Prometheus comme source de donn\'ees. Elle permet de cr\'eer des tableaux de bord interactifs avec des graphiques, jauges et alertes en temps r\'eel.

\subsection{Exporters}
\begin{itemize}
    \item \textbf{Node Exporter} : M\'etriques syst\`eme (CPU, RAM, disque, r\'eseau).
    \item \textbf{cAdvisor} : M\'etriques des conteneurs Docker (CPU, m\'emoire, I/O par conteneur).
    \item \textbf{OpenStack Exporter} : M\'etriques sp\'ecifiques OpenStack (nombre d'instances, \'etat des services).
\end{itemize}

\section{Architecture du Stockage : Cinder + LVM}

Pour le stockage bloc, un \textbf{loopback device LVM} de 20 Go est utilis\'e. Ce choix permet d'\'emuler un disque physique d\'edi\'e sans partition suppl\'ementaire :

\end{lstlisting}

\chapter*{Conclusion G\'en\'erale et Perspectives}
\addcontentsline{toc}{chapter}{Conclusion G\'en\'erale et Perspectives}

\section*{Bilan du Projet}

Ce projet a permis de d\'emontrer qu'il est possible de concevoir, d\'eployer et superviser un \textbf{Cloud Priv\'e OpenStack complet} sur une infrastructure minimale (un seul n\oe{}ud physique), gr\^ace aux outils d'automatisation modernes.

Les principaux r\'esultats obtenus sont :
\begin{enumerate}
    \item \textbf{D\'eploiement r\'eussi de 38 conteneurs Docker} via Kolla-Ansible, couvrant l'ensemble des services OpenStack (Keystone, Nova, Neutron, Glance, Cinder, Horizon, Placement).
    \item \textbf{Automatisation compl\`ete du provisionnement IaC} via Terraform : r\'eseaux virtuels, sous-r\'eseaux, routeurs, groupes de s\'ecurit\'e et instances de calcul.
    \item \textbf{Monitoring temps r\'eel} via Prometheus/Grafana avec collecte de m\'etriques syst\`eme, conteneurs Docker et services OpenStack.
    \item \textbf{Haute disponibilit\'e} assur\'ee par HAProxy/Keepalived avec gestion automatique de l'IP Virtuelle via le protocole VRRP.
\end{enumerate}

\section*{Comp\'etences Acquises}

Ce projet a permis d'acqu\'erir des comp\'etences transversales en :
\begin{itemize}
    \item \textbf{Administration Linux avanc\'ee} : Gestion LVM, configuration r\'eseau, namespaces, cgroups.
    \item \textbf{Conteneurisation} : Docker, gestion des volumes, r\'eseaux overlay.
    \item \textbf{Orchestration} : Ansible, playbooks, id empotence, gestion des r\^oles.
    \item \textbf{Infrastructure as Code} : Terraform, cycle plan-apply, gestion du state.
    \item \textbf{Monitoring et Observabilit\'e} : Prometheus, PromQL, Grafana, alerting.
    \item \textbf{SDN (Software Defined Networking)} : Open vSwitch, Neutron, DHCP, L3 routing.
\end{itemize}

\section*{Perspectives et \'Evolutions}

Plusieurs axes d'am\'elioration sont envisag\'es pour les \'evolutions futures de cette architecture :

\begin{enumerate}
    \item \textbf{Passage au multi-n\oe{}ud} : Ajout de n\oe{}uds Compute et Storage d\'edi\'es pour une v\'eritable architecture distribu\'ee.
    \item \textbf{Int\'egration de CEPH} : Remplacement du backend LVM par un syst\`eme de stockage distribu\'e CEPH pour la r\'esilience et la scalabilit\'e du stockage.
    \item \textbf{Pipeline CI/CD} : Automatisation des mises \`a jour OpenStack via GitOps (ArgoCD ou GitLab CI).
    \item \textbf{S\'ecurit\'e renforc\'ee} : Impl\'ementation de Barbican (gestion des secrets), mise en place de TLS sur tous les endpoints, et int\'egration d'un IDS/IPS.
    \item \textbf{Orchestration de conteneurs applicatifs} : D\'eploiement de Kubernetes (Magnum) au-dessus d'OpenStack pour fournir un service CaaS (Container as a Service).
    \item \textbf{Heat et Auto-Scaling} : Utilisation du service Heat pour l'orchestration de stacks et la mise \`a l'\'echelle automatique des instances.
\end{enumerate}

Ce projet confirme que la ma\^itrise du \textbf{Software Defined Data Center (SDDC)} est un atout strat\'egique dans le paysage technologique actuel. Les m\'ethodologies DevOps appliqu\'ees au Cloud Priv\'e ouvrent la voie vers des infrastructures toujours plus agiles, r\'esilientes et automatis\'ees.

#!/bin/bash
# OpenStack Private Cloud -- Kolla-Ansible Automated Setup
# Debian 12 (Bookworm) -- all-in-one setup
set -euo pipefail

KOLLA_VENV="$HOME/kolla-venv"
MANAGEMENT_INTERFACE="eth0"
EXTERNAL_INTERFACE="eth1"
VIP_ADDRESS="192.168.56.250"

# Phase 1: System packages
sudo apt-get update && sudo apt-get install -y \
  python3-dev python3-venv libffi-dev gcc libssl-dev docker.io

# Phase 2: Python virtual environment
python3 -m venv $KOLLA_VENV
source $KOLLA_VENV/bin/activate
pip install kolla-ansible

# Phase 3: Kolla configuration
sudo mkdir -p /etc/kolla
sudo chown $USER:$USER /etc/kolla
cp -r $KOLLA_VENV/share/kolla-ansible/etc_examples/kolla/* /etc/kolla/
kolla-genpwd

# Phase 4: Flush external interface (CRITICAL)
sudo ip addr flush dev $EXTERNAL_INTERFACE

# Phase 5: Create Cinder LVM loopback
sudo fallocate -l 20G /var/lib/cinder_data.img
sudo losetup /dev/loop0 /var/lib/cinder_data.img
sudo pvcreate /dev/loop0
sudo vgcreate cinder-volumes /dev/loop0

# Phase 6: Deploy
kolla-ansible bootstrap-servers -i /etc/kolla/all-in-one
kolla-ansible prechecks -i /etc/kolla/all-in-one
kolla-ansible deploy -i /etc/kolla/all-in-one
kolla-ansible post-deploy -i /etc/kolla/all-in-one

echo "Deployment complete! Access Horizon at http://$VIP_ADDRESS"
